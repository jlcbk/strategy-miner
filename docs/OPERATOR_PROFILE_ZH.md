# Operator Profile

这份文件定义“适合我们操作”的默认判断框架。策略收益再高，如果和操作条件不匹配，也不应该优先推进。

## 默认假设

- 市场：Crypto 全市场，早期优先 Binance、OKX、Bybit、Bitget。
- 产品：spot、perp、future；期权先作为后续扩展。
- 交易动作：研究和模拟优先，不自动实盘下单。
- 自动化程度：允许离线研究、回放、告警和人工确认；不默认允许无人值守执行。
- 数据偏好：优先使用可长期采集、可复验、成本可控的数据。

## 策略适配维度

每个策略进入验证前，都应补充这些判断：

- 资金规模适配：小资金、中等资金还是高容量。
- 持仓周期：秒级、分钟级、小时级、日级或更长。
- 盯盘要求：是否需要持续人工盯盘。
- 自动化要求：手动可执行、半自动可执行、必须全自动。
- 回撤和尾部风险：最大单日损失、极端行情暴露、清算风险。
- 杠杆依赖：不使用杠杆、低杠杆、中高杠杆。
- 交易所依赖：单交易所、跨交易所、是否需要多账户。
- 数据依赖：普通公开数据、订单簿高频数据、外部数据或链上数据。
- 执行复杂度：普通限价/市价、多腿同步、跨所转账或库存管理。

## 默认优先级

优先推进：

- 可以用公开数据复验的策略。
- 不依赖极低延迟竞争的策略。
- 持仓周期在分钟级到日级之间的策略。
- 成本项清晰、失败模式清晰的策略。
- 可以先用小资金模拟或人工确认的策略。

降低优先级：

- 必须全自动且延迟敏感的策略。
- 必须高杠杆或承受高尾部风险的策略。
- 依赖难以稳定获取的数据源。
- 只在极短历史窗口有效，且难以做跨市场验证。
- 执行链路包含复杂跨所转账或库存再平衡。
